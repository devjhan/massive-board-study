## 대규모 시스템 개발
### 대규모 시스템은 어떠한 기술적 · 비기술적 요구사항을 가지는가?
### 쓰기 트래픽과 읽기 트래픽이 비대칭적인 경우 어떻게 최적화하는가?
> [!NOTE] 문제 정의: MSA 환경에서의 데이터 파편화와 조회 문제
> 시나리오: 개별 게시글 조회 시 Article, Like, Comment, View 데이터가 동시에 필요하다.
> > [!CAUTION] 기존 방식의 한계
> > 1. 성능 저하: 조회 시마다 각 서비스를 동기적으로 호출하여 데이터를 조합해야 하므로 응답 시간이 최장 호출 서비스에 종속된다.
> > 2. 가용성 저하: 읽기 트래픽 폭증 시 호출 대상이 되는 모든 서비스의 부하가 동반 상승하며, 특정 서비스 장애가 게시글 조회 전체의 장애로 전파된다.
> > 3. 부하 격리 실패: 읽기 부하가 쓰기 로직의 성능과 격리되지 못한다.
> > 4. 강한 결합도: 서비스 간 순환참조 및 의존성 증가로 아키텍처의 복잡도가 증가한다.

> [!TIP] 솔루션 1. CQRS 기반의 조회 서비스 분리
> 1. 조회 전용 모델: 여러 서비스의 데이터를 비정규화하여 ArticleQueryModel로 통합하여 관리한다.
> 2. 조회 전용 서비스: 쓰기 모델과 읽기 모델을 분리하여 조회 전담 서비스를 구축한다.
> 3. Cache-Aside 전략: Redis 캐시 미스 시에만 외부 서비스를 호출해 데이터를 갱신한다.

> [!TIP] 솔루션 2. 캐싱 및 쓰기 지연 전략
> 1. 데이터 신선도 유지: 조회 빈도가 높은 데이터는 Redis에 저장하며, TTL은 24시간으로 설정하여 메모리 효율과 데이터 신선도를 유지한다.
> 2. 조회수 업데이트 최적화: 조회수와 같이 빈번한 쓰기가 발생하는 데이터는 Redis에서 먼저 처리하고, 100회 단위(BACK_UP_BATCH_SIZE)로 DB에 백업하여 저장소 부하를 분산한다.

> [!TIP] 솔루션 3: 데이터 일관성 보장
> 1. Transactional Outbox Pattern: MessageRelay를 통해 이벤트 발행과 서비스와 DB간의 트랜잭션을 원자화한다. 
> 2. Event-Driven: ArticleReadService는 Kafka로부터 ArticleCreated, CommentCreated, ArticleLiked 등의 이벤트를 수신하여 즉시 조회 모델을 최신화한다.
